# RFC: Generic Entity ID Type Support

- **RFC ID**: 2026-04-07-generic-entity-id-type
- **Status**: Draft
- **Author(s)**: Iván Falletti
- **Created**: 2026-04-07
- **Last Updated**: 2026-04-07

## Summary

This RFC proposes removing the hardcoded `int` identity type from the `EntityBase` class and making the identity type configurable at the entity level through T4 template generation. The `IEntity` interface and all repository/provider interfaces will be made generic over the identity type (`TId`). This change enables entities to use either `int` or `Guid` (or any other comparable type) as their identifier without code duplication or compatibility workarounds.

## Motivation

### Current Problems

1. **One-size-fits-all approach**: The current `EntityBase` class hardcodes `int` as the identity type, forcing all entities to use integers as their primary key. This is inflexible for projects requiring GUID-based identifiers.

2. **Code duplication**: Projects needing different identity types must either:
   - Create parallel entity hierarchies
   - Maintain separate base classes
   - Use workarounds or adapters

3. **Repository/Provider limitations**: Repository interfaces (`IReadRepository<TEntity>`, `IEditRepository<TEntity>`) hardcode `int` parameters in methods like `GetByIdAsync(int id)`. Adding GUID support requires interface variants or reflection-based approaches.

4. **Generated code inconsistency**: T4 templates cannot express identity type choice; the base class decision is fixed regardless of project needs.

### Why Now

With T4-generated entity classes, we can move identity type definition into the generation context. This eliminates the need for a universal `EntityBase.Id` property and allows projects to specify identity types per entity or per domain model.

### Expected Outcomes

- Support both `int` and `Guid` identities (and other types if needed)
- Eliminate code duplication across projects with different identity strategies
- Improve type safety with generic repository/provider interfaces
- Maintain clean architecture and separation of concerns
- Enable cleaner API contracts (type-safe vs. boxing/object-based approaches)

## Detailed Design

### 1. **IEntity Interface Changes**

**Current:**
```csharp
public interface IEntity
{
    int Id { get; }
    bool IsNew();
}
```

**Proposed:**
```csharp
public interface IEntity<TId> where TId : struct, IEquatable<TId>
{
    TId Id { get; }
    bool IsNew();
}
```

**Rationale:**
- Constraint `struct` ensures identity types are value types with consistent equality semantics
- Constraint `IEquatable<TId>` ensures proper equality checking and reliable EF Core SQL translation
- Value type constraint eliminates unreliable reference-type identities (bad practice)
- Single interface supports any comparable value type (int, Guid, long, etc.)
- Ordering capability not needed for ID operations
- No need for separate interfaces per type

### 2. **EntityBase Changes**

**Current:**
```csharp
public abstract class EntityBase : IEntity
{
    [Key]
    public int Id { get; set; }

    public bool IsNew() => Id == default;
}
```

**Proposed:**
```csharp
public abstract class EntityBase<TId> : Interfaces.IEntity<TId> where TId : struct, IEquatable<TId>
{
    /// <summary>
    /// Gets or sets the identifier.
    /// </summary>
    /// <value>
    /// The identifier.
    /// </value>
    [Key]
    public TId Id { get; set; }

    /// <summary>
    /// Determines whether this instance is new.
    /// </summary>
    /// <returns>
    ///   <c>true</c> if this instance is new; otherwise, <c>false</c>.
    /// </returns>
    public bool IsNew() => Id.Equals(default(TId));
}
```

**Rationale:**
- Generic base class eliminates hardcoded `int` type; T4 templates specify actual type via inheritance
- Concrete `Id` property works directly with any `TId` type without requiring override
- `IsNew()` uses `Equals()` for robust comparison with default value (works for any comparable type)
- No abstract members required; entities inherit fully functional implementation
- Simpler T4 template generation—just inherit from `EntityBase<TId>` without property overrides

### 3. **T4 Template Updates**

Entities generated by T4 will specify their identity type through inheritance:

```csharp
// Example: int-based entity
public partial class Product : EntityBase<int>, IProduct
{
    // Id property and IsNew() inherited from EntityBase<int>
    // No need to override—they work directly with int type

    // ... rest of generated code
}

// Example: Guid-based entity
public partial class Account : EntityBase<Guid>, IAccount
{
    // Id property and IsNew() inherited from EntityBase<Guid>
    // No need to override—they work directly with Guid type

    // ... rest of generated code
}
```

**Rationale:**
- T4 templates only specify the generic type parameter—no property overrides needed
- Eliminates repetitive property generation
- `EntityBase<TId>` provides full implementation for any comparable type
- Simpler templates reduce maintenance burden and generation complexity

### 4. **Repository Interface Changes**

**Current:**
```csharp
public interface IReadRepository<TEntity> : IRepository
    where TEntity : EntityBase
{
    Task<TEntity?> GetByIdAsync(int id);
    Task<IEnumerable<TEntity>> GetByIdsAsync(IEnumerable<int> ids);
    Task<PaginatedResultDto<TEntity>> SearchAsync<TParameters>(TParameters parameters)
        where TParameters : PaginationParametersBase;
    // ...
}
```

**Proposed:**
```csharp
public interface IReadRepository<TEntity, TId> : IRepository
    where TEntity : EntityBase<TId>
    where TId : struct, IEquatable<TId>
{
    Task<TEntity?> GetByIdAsync(TId id);
    Task<IEnumerable<TEntity>> GetByIdsAsync(IEnumerable<TId> ids);
    Task<PaginatedResultDto<TEntity>> SearchAsync<TParameters>(TParameters parameters)
        where TParameters : PaginationParametersBase;
    // ...
}

public interface IEditRepository<TEntity, TId> : IReadRepository<TEntity, TId>
    where TEntity : EntityBase<TId>
    where TId : struct, IEquatable<TId>
{
    Task<TEntity> AddAsync(TEntity entity);
    Task<TEntity> UpdateAsync(TEntity entity);
    Task DeleteAsync(TEntity entity);
    // ...
}
```

**Rationale:**
- `TId` parameter makes repositories type-safe for identity operations
- Eliminates boxing/casting concerns
- Matches entity's identity type at compile time
- Clear dependency on identity type throughout repository contract

### 4.1. **Repository Base Class Changes**

**Current:**
```csharp
public abstract class RepositoryBase<TContext> : IRepository
    where TContext : DbContextBase
{
    protected IServiceProvider ServiceProvider { get; }
    protected IUnitOfWork UnitOfWork { get; }
    protected TContext EntityContext { get; }
}

public abstract class ReadRepositoryBase<TEntity, TContext> : RepositoryBase<TContext>, IReadRepository<TEntity>
    where TEntity : EntityBase  // int-based
    where TContext : DbContextBase
{
    public virtual async Task<TEntity?> GetByIdAsync(int id)  // int hardcoded
        => await AsQueryable().FirstOrDefaultAsync(x => x.Id == id);

    public virtual async Task<IEnumerable<TEntity>> GetByIdsAsync(IEnumerable<int> ids)  // int hardcoded
        => await AsQueryable().Where(x => ids.Contains(x.Id)).ToListAsync();
}

public abstract class EditRepositoryBase<TEntity, TContext> : ReadRepositoryBase<TEntity, TContext>, IEditRepository<TEntity>
    where TEntity : EntityBase
    where TContext : DbContextBase
{
    // Inherits all methods; all work with int identity
}
```

**Proposed:**
```csharp
public abstract class RepositoryBase<TContext> : IRepository
    where TContext : DbContextBase
{
    // No changes needed; generic-agnostic infrastructure
    protected IServiceProvider ServiceProvider { get; }
    protected IUnitOfWork UnitOfWork { get; }
    protected TContext EntityContext { get; }
}

public abstract class ReadRepositoryBase<TEntity, TContext, TId> : RepositoryBase<TContext>, IReadRepository<TEntity, TId>
    where TEntity : EntityBase<TId>  // Now generic over TId
    where TContext : DbContextBase
    where TId : struct, IEquatable<TId>
{
    public virtual async Task<TEntity?> GetByIdAsync(TId id)  // Type-safe TId
        => await AsQueryable().FirstOrDefaultAsync(x => x.Id == id);

    public virtual async Task<IEnumerable<TEntity>> GetByIdsAsync(IEnumerable<TId> ids)  // Type-safe TId
        => await AsQueryable().Where(x => ids.Any(idVal => x.Id == idVal)).ToListAsync();
}

public abstract class EditRepositoryBase<TEntity, TContext, TId> : ReadRepositoryBase<TEntity, TContext, TId>, IEditRepository<TEntity, TId>
    where TEntity : EntityBase<TId>
    where TContext : DbContextBase
    where TId : struct, IEquatable<TId>
{
    // All inherited methods now work with generic TId
}
```

**Key Implementation Details:**
- `struct` constraint ensures value type identity semantics; `==` operator is always available and reliably translates to SQL equality
- `Where(x => ids.Contains(x.Id))` becomes `Where(x => ids.Any(idVal => x.Id == idVal))` for reliable EF Core translation
- Struct constraint eliminates reference-type identity issues; guarantees consistent equality behavior
- Constraint `where TEntity : EntityBase<TId>` enforces compile-time type safety

**Concrete Implementation Example:**
```csharp
// For int-based entities
public class ProductRepository : EditRepositoryBase<Product, AppDbContext, int>
{
    public ProductRepository(IServiceProvider serviceProvider) : base(serviceProvider) { }
}

// For Guid-based entities
public class AccountRepository : EditRepositoryBase<Account, AppDbContext, Guid>
{
    public AccountRepository(IServiceProvider serviceProvider) : base(serviceProvider) { }
}
```

### 5. **Provider Class Changes**

**ProviderBase Class:**
- No changes required. `ProviderBase` is infrastructure-level and doesn't depend on entity types or identity.
- It provides access to `ServiceProvider` and is inherited by all provider classes.

**ReadProviderBase & EditProviderBase Changes:**

**Current:**
```csharp
public abstract class ReadProviderBase<TInterface, TView, TViewRepository> : ProviderBase, IReadProvider<TView>
    where TInterface : Interfaces.IEntity  // Currently constrains to int Id
    where TView : EntityBase, TInterface, new()
    where TViewRepository : IReadRepository<TView>
{
    protected TViewRepository ViewRepository { get; }

    public virtual async Task<TView> GetByIdAsync(int id)  // int hardcoded
    {
        return await ViewRepository.GetByIdAsync(id) ?? throw new NotFoundException(...);
    }
}
```

**Proposed:**
```csharp
public abstract class ReadProviderBase<TInterface, TView, TViewRepository, TId> : ProviderBase, IReadProvider<TView>
    where TInterface : Interfaces.IEntity<TId>
    where TView : EntityBase<TId>, TInterface, new()
    where TViewRepository : IReadRepository<TView, TId>
    where TId : struct, IEquatable<TId>
{
    protected TViewRepository ViewRepository { get; }

    public virtual async Task<TView> GetByIdAsync(TId id)  // Type-safe TId
    {
        return await ViewRepository.GetByIdAsync(id) ?? throw new NotFoundException(...);
    }
}

public abstract class EditProviderBase<TInterface, TEntity, TView, TRepository, TViewRepository, TId>
    : ReadProviderBase<TInterface, TView, TViewRepository, TId>, IEditProvider<TView>
    where TInterface : Interfaces.IEntity<TId>
    where TEntity : EntityBase<TId>, TInterface, new()
    where TView : EntityBase<TId>, TInterface, new()
    where TRepository : IEditRepository<TEntity, TId>
    where TViewRepository : IReadRepository<TView, TId>
    where TId : struct, IEquatable<TId>
{
    protected TRepository Repository { get; }
    protected IUnitOfWork UnitOfWork { get; }

    public virtual async Task<TView> AddAsync(TView view)  // TId flows through entire hierarchy
    {
        var entity = view.MapFrom(ServiceProvider, view) ?? throw new ValidationException(...);
        var added = await Repository.AddAsync(entity);
        await UnitOfWork.CommitAsync();
        return added.MapTo(ServiceProvider);
    }
}
```

**Concrete Implementation Examples:**

```csharp
// For int-based entities
public class ProductReadProvider : ReadProviderBase<IProduct, ProductView, IProductViewRepository, int>
{
    public ProductReadProvider(IServiceProvider serviceProvider) : base(serviceProvider) { }
}

public class ProductEditProvider : EditProviderBase<IProduct, Product, ProductView, IProductRepository, IProductViewRepository, int>
{
    public ProductEditProvider(IServiceProvider serviceProvider) : base(serviceProvider) { }
}

// For Guid-based entities
public class AccountReadProvider : ReadProviderBase<IAccount, AccountView, IAccountViewRepository, Guid>
{
    public AccountReadProvider(IServiceProvider serviceProvider) : base(serviceProvider) { }
}

public class AccountEditProvider : EditProviderBase<IAccount, Account, AccountView, IAccountRepository, IAccountViewRepository, Guid>
{
    public AccountEditProvider(IServiceProvider serviceProvider) : base(serviceProvider) { }
}
```

**Rationale:**
- `TId` is explicit throughout the hierarchy (entity → repository → provider)
- Type safety enforced at compile time—repositories and entities must share same `TId`
- No casting or boxing of identity values
- Provider interface (`IReadProvider<TView>`) stays simple; `TId` is internal implementation detail
- Controllers automatically receive correct type via DI (no manual casting)
- T4 templates can generate correct provider signatures by reading entity's `TId` from base class

### 6. **Controller/API Changes**

**Approach:** No changes to controller signatures when using matching identity types. ASP.NET Core route binding handles type conversion automatically.

```csharp
// For int-based entities
[ApiController]
[Route("api/[controller]")]
public class ProductController : ControllerBase
{
    private readonly IReadProvider<ProductView> _provider;

    [HttpGet("{id}")]
    public async Task<ActionResult<ProductView>> GetById(int id)  // int from route
    {
        var result = await _provider.GetByIdAsync(id);
        return Ok(result);
    }
}

// For Guid-based entities
[ApiController]
[Route("api/[controller]")]
public class AccountController : ControllerBase
{
    private readonly IReadProvider<AccountView> _provider;

    [HttpGet("{id}")]
    public async Task<ActionResult<AccountView>> GetById(Guid id)  // Guid from route
    {
        var result = await _provider.GetByIdAsync(id);
        return Ok(result);
    }
}
```

**Rationale:**
- Controller method signature naturally reflects the entity's identity type
- Route binding automatically converts string route parameter to correct type (`int` or `Guid`)
- Type safety propagates from controller → provider → repository → entity
- No manual parsing, casting, or string-based ID handling needed
- Invalid ID types are caught at compile time (provider doesn't accept wrong type)

### 7. **Dependency Injection & Registration**

With the new generic identity types, DI container registration remains straightforward because providers are registered by their view interface (which doesn't expose `TId`):

```csharp
// In ConfigureServices or extension method
public static void AddProviders(this IServiceCollection services)
{
    // int-based providers (Product)
    services.AddScoped<IReadProvider<IProductView>, ProductReadProvider>();
    services.AddScoped<IEditProvider<IProductView>, ProductEditProvider>();

    // int-based repositories and repositories must match TId
    services.AddScoped<IReadRepository<ProductView, int>, ProductViewRepository>();
    services.AddScoped<IEditRepository<Product, int>, ProductRepository>();

    // Guid-based providers (Account)
    services.AddScoped<IReadProvider<IAccountView>, AccountReadProvider>();
    services.AddScoped<IEditProvider<IAccountView>, AccountEditProvider>();

    // Guid-based repositories must match TId
    services.AddScoped<IReadRepository<AccountView, Guid>, AccountViewRepository>();
    services.AddScoped<IEditRepository<Account, Guid>, AccountRepository>();
}
```

**Key Points:**
- Provider interface registration (`IReadProvider<TView>`) hides `TId` from consumers
- Repository registration explicitly includes `TId` type parameter
- Compiler ensures repositories are created for correct `TId`
- Dependency injection automatically provides correctly-typed repository to provider
- No need for factory patterns or reflection-based resolution

### 8. **Backward Compatibility Strategy**

**Breaking Change - No Compatibility Layer**

This RFC intentionally introduces breaking changes to eliminate code duplication and establish a clean, type-safe architecture across all projects. All projects using these base classes will be updated to the new generic approach:

**Affected components (all updated):**
- `EntityBase<TId>` replaces `EntityBase`
- `IEntity<TId>` replaces `IEntity`
- `IReadRepository<TEntity, TId>` replaces `IReadRepository<TEntity>`
- `IEditRepository<TEntity, TId>` replaces `IEditRepository<TEntity>`
- `ReadProviderBase<..., TId>` replaces `ReadProviderBase<...>`
- `EditProviderBase<..., TId>` replaces `EditProviderBase<...>`
- All repository implementations updated
- All provider implementations updated
- All entity implementations regenerated via T4
- All controller signatures updated where ID parameters appear
- All tests updated

**Code migration is straightforward** because:
1. T4 entity regeneration is automated
2. Repository/provider changes follow a consistent pattern
3. Controller changes are mostly method signature updates
4. IDE refactoring tools can automate most updates (find & replace patterns, rename refactoring)

No compatibility adapters or deprecation periods required. This is a clean break enabling better architecture going forward.

## Alternatives Considered

### Alternative 1: Keep EntityBase as-is, add IEntity<TId> in parallel
- **Approach**: Introduce new `IEntity<TId>` interface; keep `IEntity` for backward compatibility
- **Advantages**: No breaking changes; can migrate gradually
- **Disadvantages**: Induces code duplication; two parallel hierarchies; confusing for developers; maintenance burden
- **Rejected**: Too much legacy code; violates DRY principle

### Alternative 2: Use object/dynamic for identity
- **Approach**: Store Id as `object` or `dynamic` to support multiple types
- **Advantages**: Single interface; no generics needed
- **Disadvantages**: Loss of type safety; boxing/unboxing overhead; harder debugging; violates CS-009 (no dynamic)
- **Rejected**: Violates type safety and coding standards

### Alternative 3: Keep int as default, add GenericEntityBase<TId> for advanced projects
- **Approach**: MainEntityBase keeps int; advanced projects use EntityBase<TId>
- **Advantages**: Avoids current friction for int-only projects
- **Disadvantages**: Two code paths; confusion about which to use; code duplication; inconsistent APIs
- **Rejected**: Creates maintenance burden; architecture should be uniform

### Alternative 4: Use reflection-based identity type discovery
- **Approach**: Keep EntityBase as-is; use reflection to determine actual identity type at runtime
- **Advantages**: Minimal interface changes
- **Disadvantages**: Runtime overhead; less type-safe; harder to debug; violates CS-007 (reflection forbidden)
- **Rejected**: Violates coding standards and reduces performance

## Testing Strategy

Since this is a breaking change with no backward compatibility path, testing focuses on validating correct behavior of the new implementation for both `int` and `Guid` identity types.

### Unit Tests

1. **Entity Tests**
   - Verify `IsNew()` returns true when `Id.Equals(default(TId))`
   - Verify `IsNew()` returns false when `Id != default(TId)`
   - Test both int and Guid entity implementations
   - Verify XML documentation is preserved

2. **Repository Tests**
   - Test `GetByIdAsync(TId id)` with int entities
   - Test `GetByIdAsync(TId id)` with Guid entities
   - Test `GetByIdsAsync(IEnumerable<TId> ids)` for both types
   - Verify compile-time type safety (compile errors when TId mismatched)
   - Test search/pagination with both entity types

3. **Provider Tests**
   - Verify providers correctly pass identity to repositories
   - Test both read and edit providers with int entities
   - Test both read and edit providers with Guid entities
   - Verify type safety in provider-to-repository calls

4. **Mapping Tests**
   - Entity ↔ DTO mapping preserves identity correctly
   - View projection maintains identity type
   - Test with both int and Guid entities

### Integration Tests

1. **End-to-end flows**
   - Complete CRUD for int-based entities
   - Complete CRUD for Guid-based entities
   - Search and pagination for both types
   - Unit of work and transaction handling

2. **API Tests**
   - Route binding for int route parameters
   - Route binding for Guid route parameters
   - HTTP 404 responses for not-found identities
   - Create and update operations with identity generation

3. **Mixed-type scenarios**
   - Multiple entities with different identity types in single API
   - Queries across different entity types
   - Correct repository/provider resolution in DI container

### Manual Testing Procedures

1. Run full test suite for framework and all projects
2. Test Example application with both int and Guid entities
3. Verify route binding works correctly for both types
4. Check error handling for invalid identity values
5. Confirm swagger/API documentation shows correct types

1. **Identity types beyond int and Guid** ✓

   - No additional restrictions needed

2. **Identity generation strategy** ✓
   - Use EF Core defaults (database identity for int, client-generated for Guid)
   - No special handling needed in framework

3. **Parameterless constructor requirement** ✓
   - No explicit constraint; rely on EF Core conventions
   - Document in interfaces that EF Core requires parameterless constructors
